# 08 Architectural Decision Records (ADR) - 关键决策记录

本文件用于记录项目演进过程中涉及的重大技术/产品决策（ADR 风格），便于新加入成员快速理解“为什么这么做”（Why），而不仅是“做了什么”（What）。

---

## ADR-001: 采用 `.prompts/` 作为工作区级存储规范
*   **日期**：2026-03-13
*   **状态**：已接受
*   **背景**：在早期三份设计书（Gemini的`.promptcraft`, Grok的`.vscode/prompts.json`，Haiku的散装YAML）中，关于 Prompt 的存储方式存在分歧。
*   **决策**：使用项目根目录下的 `.prompts/` 文件夹存储分离的 `.yaml` 文件，而非单体 JSON 或者本地 SQLite 数据库。
*   **理由**：
    1.  **版本控制友好**：细粒度的 YAML 文件能直接在 Git diff 中清晰展示 Prompt 的修改历史。
    2.  **团队协作**：开发者可以像 Review 代码一样 Review `.prompts/` 里的修改。
    3.  **人类可读性**：多行 Prompt（包含大量的 Markdown 或代码片段）在 JSON 中需要转义，难以阅读；YAML 支持多行字面量，完美适配这一需求。

## ADR-002: 全面更名为 "Prompt by Prompt" (PbP)
*   **日期**：2026-03-13
*   **状态**：已接受
*   **背景**：设计初期有 `PromptCraft`, `PromptForge`, `PromptFlow` 三个内部代号。
*   **决策**：在重构文档与确立最终产品愿景时，采用更加口语化且富有工程哲学的名称 "Prompt by Prompt"（简称 PbP）。
*   **理由**：
    1.  更生动表达了“步步为营、精细打磨”的 Prompt 迭代过程。
    2.  避免了与其他知名工具（如微软官方的 Prompt flow）产生直接的商标或认知冲突。

## ADR-003: 放弃内置大型对话机器人的目标
*   **日期**：2026-03-13
*   **状态**：已接受
*   **背景**：市面上如 Continue, GitHub Copilot Chat 等均提供了全局对话框。
*   **决策**：PbP 不做类似 ChatGPT 侧边栏的通用闲聊工具。
*   **理由**：
    1.  **极简定位**：我们只专注于“特定任务的流水线执行”（如选中一段代码 -> 右键 -> 生成单测 -> 插入文件）。
    2.  **避免沉重**：聊天界面涉及复杂的对话历史管理、Markdown 渲染、多轮上下文拼接，这会显著增加插件体积并偏离“管理模板”的核心目标。

## ADR-004: 第一公民支持本地模型 (Ollama)
*   **日期**：2026-03-13
*   **状态**：已接受
*   **背景**：许多企业开发者受限于安全合规，无法将源码发送至 OpenAI 或 Anthropic。
*   **决策**：MVP 阶段即接入并验证 Ollama 支持。
*   **理由**：
    1.  主打“隐私安全”、“0 成本配置”。
    2.  这正是 PbP 吸引早期极客用户的巨大卖点（因为竞品多强制绑定云端 API 甚至强制登录账户）。
